# Versioning Policy

This document is the canonical versioning policy for `langgraph-forge`.
It is self-contained: read it without `MAINTAINERS.md` open and you have
everything you need to predict, plan, and consume releases.

## A. Scheme

[Semantic Versioning 2.0](https://semver.org/) with [PEP 440 pre-release
identifiers](https://peps.python.org/pep-0440/#pre-releases):

```
MAJOR.MINOR.PATCH[<a|b|rc><N>]
```

Examples on the projected release path:

```
0.1.0a1   →  0.1.0a2  →  0.1.0rc1  →  0.1.0  →  0.1.1  →  0.2.0
```

Stable versions never carry a pre-release suffix. `pip install
langgraph-forge` resolves only to stable; pre-releases require
`pip install --pre langgraph-forge` or an explicit version pin.

## B. Pre-1.0 semantics (where we are)

- **MINOR bump (`0.x → 0.x+1`)** ships when **any** breaking change
  lands to a publicly-exported symbol — i.e., anything reachable
  through `import langgraph_forge` (the names in
  `src/langgraph_forge/__init__.py`'s `__all__`), the
  `langgraph-forge` CLI surface, or the `DeploymentAdapter` Protocol
  contract.
- **PATCH bump (`0.x.y → 0.x.y+1`)** ships for bug fixes, internal
  refactors, doc-only changes, and dependency-pin tightening that does
  not exclude a previously resolvable version.
- **MAJOR stays at `0`** until the criteria in § D are met.

Pre-1.0 minor versions carry no compatibility promise beyond what the
release notes explicitly call out. Users who need stability should pin
to an exact version (`langgraph-forge == 0.1.0`) and upgrade
deliberately.

## C. Pre-release identifiers

| Suffix | Stage | Meaning |
|---|---|---|
| `aN` | alpha | API may change between alphas. CI must be green; published to PyPI; `pip` requires `--pre`. |
| `bN` | beta | API frozen for the cycle barring critical issues. Skippable if alpha proved stable. |
| `rcN` | release candidate | No further changes intended; only RC-level bug fixes accepted. |
| (none) | stable | Final release for that MINOR.PATCH. |

We do not use post-releases (`.post1`) or dev-releases (`.devN`).

## D. Path to 1.0

The `0` MAJOR is dropped only when **all three** gates pass:

1. The `DeploymentAdapter` Protocol has shipped without a breaking
   change for **≥ 60 days**.
2. **At least three independent deployment adapters** exist — in-tree
   counts (currently `direct`, plus `bedrock` / `vertex` / `azure`
   stubs that need to graduate to working implementations); third-party
   adapters published via the `langgraph_forge.deployment_adapters`
   entry-point group also count.
3. **Documentation covers every Diátaxis quadrant**: at least one
   tutorial, the four most-asked how-to guides, complete API reference
   for the public surface, and at least one explanation page per major
   architectural choice.

Until these three are met, the project remains in `0.x` even if it
feels stable in practice. Conversely, meeting them does not force a
1.0 release — they are necessary, not sufficient. The maintainer
decides when 1.0 ships.

## E. Post-1.0 semantics (for future reference)

- **MAJOR**: any breaking public-API change. Always paired with a
  formal RFC issue and **≥ 1 month notice** announced on the previous
  minor's release notes and pinned issue.
- **MINOR**: backward-compatible features.
- **PATCH**: backward-compatible fixes.

Breaking change means: a previously valid call to a public symbol now
errors, returns a different type, has different observable side
effects, or requires a different argument shape. Internal symbols
(modules under `langgraph_forge` not re-exported through
`__init__.py`) are not part of the contract and may change without
notice.

## F. Deprecation

When a public symbol must be removed, the staged retirement is:

1. **Land replacement.** A `feat` or `refactor` PR adds the new
   symbol. The old symbol stays.
2. **Warn at use site.** The next PR adds a `DeprecationWarning`
   (or `FutureWarning` for end-user-facing surfaces) on first use of
   the old symbol. The warning message names the replacement and the
   release that will remove it.
3. **Document.** The deprecation appears in the GitHub Release notes
   that ship the warning, under a `### Deprecated` section
   auto-generated by `git-cliff` from `refactor!:` commits or
   `BREAKING CHANGE:` footers.
4. **Wait.**
   - **Pre-1.0**: leave the deprecated symbol in place for **at least
     2 minor cycles** (e.g., deprecated in `0.3.0` → removable
     starting `0.5.0`).
   - **Post-1.0**: same 2-minor-cycle wait **and** removal must
     coincide with a major bump.
5. **Remove.** A `feat!: ...` or `refactor!: ...` commit drops the
   old symbol. Pre-1.0 the `!` signals "breaking-in-minor"; post-1.0
   it triggers the major bump.

Skipping the warn step (jumping from "land replacement" straight to
"remove") is forbidden — it eliminates the consumer's signal that they
need to migrate.

## G. Yank vs delete

A version published to PyPI is **yanked**, never deleted, when it is
discovered to be:

- A **security risk** (CVE-worthy or credential-leaking).
- **Data-corrupting** to consumers of the API.
- **Install-breaking** (does not produce a working `import
  langgraph_forge`).

Yanking happens at <https://pypi.org/manage/project/langgraph-forge/>:
the version is hidden from `pip install` resolution but stays
downloadable so users who pinned it (`langgraph-forge == 0.1.4`)
continue to function. **Never delete a published version** — `pip`
caches reference the file hash; deletion breaks downstream pins and is
treated as a supply-chain incident.

The corresponding GitHub Release stays published. Edit it to add a
prominent note linking to the fix release.

## H. Where the canonical source lives

| Surface | Location | Authority |
|---|---|---|
| Current development version | `src/langgraph_forge/_version.py` (single line) | Read by hatchling at build time via `[tool.hatch.version]` in `pyproject.toml`. The runtime value of `langgraph_forge.__version__` mirrors this. |
| Released versions | Tags on `main` (`v0.1.0a1`, `v0.1.0`, …) | Tag pattern enforced by `cliff.toml`'s `tag_pattern`; pushing a `v*` tag triggers `.github/workflows/publish.yml`. |
| Release notes | GitHub Releases | Generated by `git-cliff --latest` per tag; bodies live in the GitHub Release UI, not in a tracked file. |
| Versioning policy (this doc) | `/VERSIONING.md` | The doc you are reading. |

There is no `CHANGELOG.md` in the repository. Decision: ADR-0003.
